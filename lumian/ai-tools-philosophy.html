02
core philosophy

Think Like Dynamic Programming

  • DP = solve each subproblem ONCE, memoize, never recompute
  • Same principle: every workflow solved once should be cached as a tool, doc, or agent
  • Without memoization (docs, APIs, shared context) - every new person, every new agent recomputes from scratch
  • With memoization - each solved problem becomes a building block. Value compounds exponentially.
naive recursion = O(2^n)
with memoization = O(n)

data strategy

Context is King

  • Data + Iteration = great outcomes for customers
  • Two hard parts: (1) having the right data, (2) how frequently we can access it
  • BigQuery = unlimited access (our own). Amazon API = rate limited but generous
  • Three data sources: Amazon account data (BQ), client interactions (Zep), market research (SmartScout)
"If we have the right data, we can iterate and reach a very good outcome for the customer."

  • Example: @lumian.ai + ASIN triggers Rufus agent in background, returns results in Slack

06
composability

CLI/API for Every Agent

  • Problem: code deployed on different VMs, no programmatic way to access other team's agents
  • For my own product - built CLI first, then integrated with the agent
  • Ask: whatever team is building, also build a tool/API so agents can access it seamlessly
  • Agents composing with agents is how we get compounding value
"Whatever they are building, just build a tool so the agent can access it seamlessly."
07
documentation

Compounding Engineering via Docs

  • Amazon has finite edge cases - all solvable if documented
  • More than humans, it helps the AGENTS - they won't fall into same loops
  • Example: if docs existed when BQ work was done, my agent wouldn't have repeated the same work
  • Problem today: 30 scattered repos, no shared documentation
"More than two humans, it will help the agents. Agents will not fall into the same loop."
08
product

Build Only What They're Dying For

  • Talk to each account management person individually
  • Find their BURNING pain point - only then they will use what we build
  • If they feel the pain, they use the tool. If we push it, they won't.
"It should be their burning pain point - only then they will use it if we build it."
09
automation

Reduce Approval Loops

  • Current state: lots of manual approval steps blocking flow
  • Goal: agent does work, drafts change, person approves, auto-applies to Amazon
  • Flip the default: agent acts, human vetoes only if needed
  • That's when we're truly AI-native
"Reduce approval to not-approving. Agent acts, human vetoes only if needed."

observability

Agent Observability

  • Expose agent metrics on dashboard: edits made, issues flagged, reviews removed
  • Show value: Sankey chart from top-line to CM2, with agent contributions highlighted
  • Builds trust and proves ROI to leadership and clients
"They can see the AI doing the magic - which AI agent flagged this, why, and what improved."
11
infrastructure

Infrastructure Hygiene

Single Server

All products are lightweight - one server can handle them. Single URL for access.

Docker

Containerize everything for easy stop/run/manage. Reproducible deploys.

Centralized Logs

OpenSearch so agents (and humans) can see all logs in one place.

API + Workers

API layer always live. Heavy pipelines run in background workers.
