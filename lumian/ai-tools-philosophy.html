AI Tools & Approach
Week 1-2 Summary

Points I've raised across team meetings since joining

Tejesh Vaish / Apr 27 - May 5, 2026

02

Context is King

  • Data + Iteration = great outcomes for customers
  • Two hard parts: (1) having the right data, (2) how frequently we can access it
  • BigQuery = unlimited access (our own). Amazon API = rate limited but generous
  • Three data sources: Amazon account data (BQ), client interactions (Zep), market research (SmartScout)
"If we have the right data, we can iterate and reach a very good outcome for the customer."
03

I Already Live This Way

  • Built my own Amazon shopping agent - opens browser, searches multiple products, reads reviews, tells me the best option
  • The agentic approach isn't theoretical for me - it's how I work daily
  • Same philosophy applies to everything we build here
"Doing something manually is the worst way to do it."
04

Unified Agent Interface via Slack

  • Slack as the single interface for all agents - meet users where they are
  • Example: @lumian.ai + ASIN triggers Rufus agent in background, returns results in Slack
  • Track usage/adoption natively - see who's using what and how
  • One link to full web UI if they need deeper interaction
"80% of workflows can be met in Slack only."
05

CLI/API for Every Agent

  • Problem: code deployed on different VMs, no programmatic way to access other team's agents
  • For my own product - built CLI first, then integrated with the agent
  • Ask: whatever team is building, also build a tool/API so agents can access it seamlessly
  • Agents composing with agents is how we get compounding value
"For others I would need help from the team - whatever they are building, just build a tool so the agent can access it seamlessly."
06

Compounding Engineering via Documentation

  • Amazon has finite edge cases - all solvable if documented
  • More than humans, it helps the AGENTS - they won't fall into same loops
  • Example: if docs existed when BQ work was done, my agent wouldn't have repeated the same work
  • Problem today: 30 scattered repos, no shared documentation
"The reason - more than two humans, it will help the agents. Agents will not fall into the same loop."
07

Memory Architecture - Deprioritized

  • Knowledge graph = way for AI to have infinite memory across all company data
  • Important but premature - we don't have any product dialed together yet
  • Priority: build something team is actually using first, then layer memory on top
"We were focusing a lot on memory but right now that is not important because we don't have any product dialed together."
08

Build Only What They're Dying For

  • Talk to each account management person individually
  • Find their BURNING pain point - only then they will use what we build
  • If they feel the pain, they use the tool. If we push it, they won't.
"It should be their burning pain point - only then they will use it if we build it."
09

Reduce Approval Loops - True Automation

  • Current state: lots of manual approval steps blocking flow
  • Goal: agent does work, drafts change, person approves, auto-applies to Amazon
  • Flip the default: agent acts, human vetoes only if needed
  • That's when we're truly AI-native
"Reduce approval to not-approving. Agent acts, human vetoes only if needed."
10

Agent Observability

  • Expose agent metrics on dashboard: edits made, issues flagged, reviews removed
  • Show value: Sankey chart from top-line to CM2, with agent contributions highlighted
  • Builds trust and proves ROI to leadership and clients
"They can see the AI doing the magic - which AI agent flagged this, why, and what improved."
11

Infrastructure Hygiene

single server

All products are lightweight - one server can handle them. Single URL for access.

docker

Containerize everything for easy stop/run/manage. Reproducible deploys.

centralized logs

OpenSearch so agents (and humans) can see all logs in one place.

api + workers

API layer always live. Heavy pipelines run in background workers.

12

Littebird - Real Workflow Data

  • Tool that screenshots your screen periodically, builds memory of what you did
  • Daily/weekly summaries of actual work - no self-reporting bias
  • For us: get TRUE picture of what team does vs what they say they do
  • Verify if they're actually using the tools we built
"Real data without any filter."
13

GPT for Classification

  • Use GPT-4 to classify Granola recordings (client calls vs personal) automatically
  • Don't need manual folder sorting - define classes, let model classify
  • Same approach for email ingestion - tag by client automatically
  • Remove manual categorization from every data pipeline
14

What's Next

  • Talk to Todd - where is he blocked? What does he do manually?
  • Build one small workflow end-to-end, prove value, then expand
  • Design system + components library for consistent UI across all tools

Build what they need. Prove it works. Then scale.
